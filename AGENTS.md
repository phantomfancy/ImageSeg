## 2. 项目目标

本仓库将演进为一个基于 `ASP.NET Core 10 + Blazor Web App + Interactive Auto + PWA + Aspire 13 + Podman` 的渐进式 Web 应用。

应用名称：

- `4C-ai装备识别工具`

目标定位：

- 面向图像/视频/摄像头输入的装备识别网页
- 既支持本地侧识别，也尽量支持服务器侧识别
- 以识别能力为核心，前端上传、采集、结果展示、导出都服务于识别流程

## 3. 已确认的业务需求

以下需求来自用户输入，属于当前项目范围：

1. 导入单张图片
2. 导入多张图片
3. 导入单个视频
4. 导入多个视频
5. 调用摄像头读取视频流
6. 摄像头校准
7. 识别功能
8. 至少支持本地识别
9. 最好支持服务器识别 + 本地识别双端识别
10. 导出结果图像
11. 导出结果视频

补充目标：

- 计划使用 `TensorFlow.NET` 进行模型训练与服务器侧推理探索
- 计划采集多张图像训练模型，对若干特定装备（物体）进行识别
- 双端识别思路可参考 `VERT` 项目，即：
  - 轻量或隐私敏感流程尽量本地完成
  - 资源消耗较高或浏览器不适合的流程可回退到服务器

## 4. 当前阶段的技术判断

以下内容是为了后续开发统一方向而给出的工程判断，属于“当前推荐方案”。如果后续验证结果不同，可以修改，但必须同步更新本文件。

### 4.1 关于 Blazor Web App 与 Auto

- 当前仓库已经是 `Blazor Web App`，但不是传统意义上的 `Blazor Server` 应用。
- 目前只是 `Blazor Web App` 中的 `Interactive Server` 渲染模式。
- 后续目标应是继续保留 `Blazor Web App` 应用模型，并把部分或全部交互组件演进到 `Interactive Auto`。
- 如果要实现 `Interactive Auto`，应按 Blazor Web App 的标准结构补充独立的 `.Client` 项目。
- PWA 的离线缓存、安装能力、前端本地计算能力，应优先围绕 `.Client` 侧设计。

### 4.2 关于 PWA

- 本项目目标不是“网页加壳”，而是真正可安装、可缓存静态资源、可在弱网下保留基础能力的 PWA。
- 首选保证：
  - 安装能力
  - 资源缓存
  - 基础页面离线可访问
  - 本地图片识别在已缓存模型和资源的前提下尽量可离线运行
- 不应默认承诺“所有能力都离线可用”。
- 视频识别、服务器识别、模型下载、批量导出等高负载功能，允许在联网条件下运行。

### 4.3 关于识别任务定义

- 当前工作假设：用户需要的是“目标装备识别”，并且结果需要叠加到图像/视频上导出。
- 因此默认按“目标检测/定位类任务”理解，而不是仅做整张图单标签分类。
- 如果后续确认只需要整图分类，可以简化模型与导出链路；确认后必须更新本文件。

### 4.4 关于 TensorFlow.NET 的使用边界

- `TensorFlow.NET` 当前优先用于：
  - 训练流程
  - 服务器侧推理
  - 离线数据处理或模型转换工具
- 当前已知外部事实：
  - `TensorFlow.NET` 官方仓库当前维护活跃度有限
  - 其常见用法依赖本机或容器内的 TensorFlow 原生运行时分发包
- 不要默认把 `TensorFlow.NET` 作为浏览器内本地推理方案。
- 当前工程假设：浏览器内本地推理更可能需要基于浏览器可运行的推理引擎，通过 JS interop 接入。
- 只有在明确验证 `TensorFlow.NET` 可稳定运行于目标浏览器执行环境后，才允许把它作为前端本地推理主方案写入实现。

### 4.5 关于双端识别策略

参考 `VERT` 的思路，本项目优先采用“双路径”设计：

- 本地识别路径：
  - 优先处理单张图片、小批量图片、低延迟预览
  - 优先满足隐私、本地离线、弱网可用
- 服务器识别路径：
  - 优先处理视频、批量任务、重模型、长时推理、导出视频
  - 允许使用容器化服务与硬件加速

当前不建议把所有识别能力都强压到浏览器端。

### 4.6 关于训练与推理的职责拆分

- 训练流程与在线推理流程应分离。
- 在线 Web 应用的重点是：
  - 导入
  - 预处理
  - 调用本地或服务器识别
  - 结果展示
  - 导出
- 模型训练、数据集整理、标注转换、模型导出、模型版本管理，应设计为独立流程或独立工具，不应直接耦合到主站页面交互里。

### 4.7 关于视频处理

- 视频处理是高风险、高资源占用模块。
- 当前默认判断：
  - 图片识别应先落地
  - 视频识别与视频导出应在后续阶段实现
- 如果浏览器端视频处理性能不足，允许采用服务器处理或本地守护进程/服务配合的方案。

### 4.8 关于摄像头与校准

- 摄像头视频流读取必须通过浏览器标准媒体能力和 JS interop 实现。
- 摄像头校准应独立为单独模块，不要把校准逻辑散落到识别页面中。
- 校准参数应可保存、命名、复用，并与具体设备或配置关联。

## 5. 推荐的目标项目结构

在不做大拆大改的前提下，优先沿现有 Aspire 结构演进：

- `4CImageSeg.AppHost`
  - 保留为 Aspire 编排入口
  - 统一管理 Web、API、模型服务、可选容器资源
- `4CImageSeg.Web`
  - 作为 ASP.NET Core 10 `Blazor Web App` 宿主
  - 负责 SSR 首屏输出，以及 `Interactive Server` / `Interactive WebAssembly` / `Auto` 组件承载
- `4CImageSeg.Web.Client`
  - 新增
  - 承担 Interactive WebAssembly / Interactive Auto / PWA / 本地推理入口
- `4CImageSeg.ApiService`
  - 可演进为识别 API
  - 若职责膨胀，可后续拆为 `Inference.Api`、`Training.Api`、`Media.Api` 等，但前期不要过度拆分
- `4CImageSeg.ServiceDefaults`
  - 继续保留
- `4CImageSeg.Tests`
  - 保留 Aspire 集成测试
  - 后续补充前端冒烟测试、API 合约测试、识别结果回归测试

## 7. 部署与运行约束

- 部署目标明确包含 `Aspire 13 + Podman`。
- 默认不要写死 Docker 专用方案。
- 涉及容器运行时时，应优先兼容 Podman。
- 若本地 Aspire 使用 Podman，相关环境变量、脚本、文档必须同步说明。
- 涉及 GPU、视频编解码、模型运行时时，必须区分：
  - 开发机 Windows + Podman 场景
  - 目标部署机 Linux/容器场景
- 未验证前，不要承诺 GPU 透传、硬件编解码、浏览器端大模型实时推理一定可用。

## 8. 编码与设计约束

- 保持小步修改，优先演进现有结构，不做无理由重命名或大搬家。
- 先删除示例业务，再引入真实业务，不要让天气示例代码长期混入正式代码。
- 新增功能时，优先建立清晰的领域边界：
  - 媒体输入
  - 校准
  - 推理
  - 结果渲染
  - 导出
  - 模型/配置管理
- 浏览器能力相关逻辑优先放在前端明确的模块中，不要把大量前端设备控制细节塞进后端。
- 高开销任务不要直接堵塞交互 UI；应支持后台任务、进度反馈、取消。
- 涉及模型、媒体文件、导出文件时，必须明确：
  - 存储位置
  - 生命周期
  - 命名规则
  - 清理策略

## 9. 测试与验证要求

- 现有测试基线是 Aspire 集成冒烟测试。
- 每次引入关键功能后，应至少补足对应的最小验证：
  - 页面可打开
  - API 可调用
  - 导入流程可走通
  - 推理结果结构正确
  - 导出文件可生成
- 与图像识别相关的测试，后续应逐步建立固定样例集，避免模型或预处理修改后结果无回归约束。
- 涉及摄像头、PWA、浏览器本地推理的功能，若自动化测试不足，必须提供明确的手工验证步骤。
- 当前已确认：
  - 用户已在本机 PowerShell 中执行 `dotnet build .\4CImageSeg.slnx`，并成功生成全部项目
  - 使用 Visual Studio / MSBuild 可正常生成并运行当前解决方案
  - 本机 CLI 与 VS/MSBuild 当前都可正常生成该解决方案
  - 某些受限沙箱中的 `dotnet build` 可能在 `4CImageSeg.Web.Client` 的 WebAssembly SDK 任务 `ComputeWasmBuildAssets` 阶段失败
  - 上述沙箱 CLI 失败当前判断为环境侧 task host 问题，不等同于当前仓库存在 C# 编译错误
  - 已使用全新官方 `Blazor Web App (Auto)` 最小模板在相同受限 CLI 环境中复现同类失败
  - 因此当前应把该问题视为“特定受限执行环境中的 `.NET SDK/MSBuild/WebAssembly task host` 组合问题”，而不是本仓库特有配置错误

## 11. 当前已知风险与待确认项

以下事项后续要优先确认，否则会影响设计路线：

- 本地识别是否要求真正纯浏览器离线运行
- 本地识别是否必须坚持 `TensorFlow.NET`
- `TensorFlow.NET` 是否能接受其当前维护状态与原生运行时依赖
- 识别任务到底是整图分类、目标检测，还是实例分割
- 视频导出是否要求浏览器本地完成，还是允许服务器端生成
- 摄像头校准的精度目标、标定流程、标定板方案
- 是否需要用户账户、历史记录、结果归档、权限隔离

如上述任一项被确认，应第一时间更新本文件。
