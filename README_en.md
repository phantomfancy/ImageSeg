# ImageSeg

[中文 README](./README.md)

`ImageSeg` is an ONNX-based visual inference workbench built with `React 19 + TypeScript + ONNX Runtime Web/WebGPU + Aspire TypeScript AppHost`. It supports image inference with Ultralytics-style and Hugging Face-style models, with a current focus on browser-side inference workflows.

## Repository Structure

- `imageseg-webapp/apphost.ts`: Aspire TypeScript AppHost implementation entry
- `imageseg-webapp/contracts`: pure TypeScript model contract package
- `imageseg-webapp/frontend`: React 19 + TS/TSX frontend
- `imageseg-training/`: training scripts, datasets, and exported models

## Common Commands

Install dependencies:

```powershell
npm install
```

Or with Bun:

```powershell
bun install
```

`aspire run` loads `imageseg-webapp/apphost.ts` through the root `aspire.config.json`, so the root `node_modules` directory must contain the AppHost runtime dependencies. If you see `Cannot find package 'vscode-jsonrpc' imported from .modules\\transport.ts`, run `npm install` or `bun install` again.

Generate the Aspire TypeScript SDK:

```powershell
aspire restore
```

If the `packages` section in `aspire.config.json` changes, run `aspire restore` again to refresh the TypeScript AppHost SDK in `.modules/`.

Start the development environment:

```powershell
npm run dev
```

Or:

```powershell
bun run dev
```

The root `npm run dev` command launches the full Aspire orchestration through `aspire run`, not Vite directly.

If you only want to start the frontend Vite dev server, use either of the following:

```powershell
cd .\imageseg-webapp\frontend
npm run dev
```

```powershell
npm run dev --workspace @4cimageseg/frontend
```

```powershell
cd .\imageseg-webapp\frontend
bun run dev
```

By default, the Vite frontend listens on `0.0.0.0:5173`, and the local access URL is typically `http://localhost:5173`. To change the port, set the `PORT` environment variable first:

```powershell
$env:PORT=3000
npm run dev --workspace @4cimageseg/frontend
```

```powershell
$env:PORT=3000
cd .\imageseg-webapp\frontend
bun run dev
```

Publish deployment artifacts:

```powershell
npm run publish
```

`npm run publish` generates `aspire-output/` in the repository root, including at least:

- `docker-compose.yaml`
- `.env`

To generate environment-specific deployment parameters, use:

```powershell
aspire do prepare-env --environment Staging
```

Deploy locally:

```powershell
npm run deploy
```

Stop and clean up the local Docker Compose deployment:

```powershell
aspire do docker-compose-down-env
```

Deployment prerequisites:

- Docker Desktop is installed and running
- `npm install` has been executed in the repository root
- `aspire restore` has been executed

If you use Bun for dependency installation, `bun install` is equivalent to the `npm install` prerequisite above.

About Podman:

- The officially supported and validated built-in path in this repository is Docker Compose.
- If you mainly use Podman locally, prefer running `npm run publish` first and then consuming the generated files under `aspire-output/` manually. This repository does not currently guarantee automatic Podman compatibility for `npm run deploy`.

## Static Site Container Deployment (Podman)

The current `addViteApp(...)` setup in `imageseg-webapp/apphost.ts` is still mainly intended for development-time orchestration. If your goal is to deploy the frontend as a standalone static site first, prefer `imageseg-webapp/frontend/Dockerfile` with an Nginx container.

Prerequisites:

- `podman machine` is running and `podman info` returns successfully
- `npm install` has been executed in the repository root

If Podman is still in `Currently starting` state or the socket is refusing connections, run:

```powershell
wsl --terminate podman-machine-default
podman machine start
podman info
```

Build the static image:

```powershell
npm run static:image
```

Start the local static site container:

```powershell
npm run static:run
```

Default access URL:

- `http://localhost:8080/`

Check whether the ONNX Runtime static assets are available:

- `http://localhost:8080/ort/ort-wasm-simd-threaded.asyncify.wasm`

Stop the container:

```powershell
npm run static:stop
```

Notes:

- The current static container deployment serves content from the root path `/`.
- ONNX model files are still imported locally by the browser and are not packaged into the image.
- If you need external publishing through HTTPS or a reverse proxy, extend the Nginx configuration on top of this setup.

Build:

```powershell
npm run build
```

Validate:

```powershell
npm run test
```

Important notes:

- `imageseg-webapp/frontend/public/ort` is generated by copying assets from `onnxruntime-web` through `npm run prepare:ort`, and it should not be committed.
- `bun.lock` is Bun's lockfile and should be committed to the repository; this repository intentionally keeps `package-lock.json` as well for npm compatibility.
- `npm run test`, including `npm run test:frontend` and `npm run verify:training-models`, depends on the test model set under `imageseg-training/training_result`. If that directory is not prepared on the current machine, these checks may fail because `.onnx`, `config.json`, or `preprocessor_config.json` files are missing. This is expected and does not necessarily indicate a problem in the frontend type checking, `tsgo` migration, or build chain itself.
- `npm run publish` and `npm run deploy` depend on the Docker Compose deployment environment declared in `imageseg-webapp/apphost.ts` and on `Aspire.Hosting.Docker` being included in `aspire.config.json`.
- If the ONNX Runtime assets under `public/ort/` are currently held open by a dev server or browser process, rerunning `npm run build` / `npm run test` or the Bun equivalents may fail while copying `ort-wasm-simd-threaded.asyncify.wasm`; stop the process holding the file and retry.
- If the training artifacts are not available, you can still validate the parts that do not depend on them with:

```powershell
npm run typecheck
npm run build
npm run test:contracts
```

Or with Bun:

```powershell
bun run typecheck
bun run build
bun run test:contracts
```

## fp16 Model Compatibility on WebGPU

Some `hf-detr-like` `fp16` ONNX models may fail under `onnxruntime-web + WebGPU` with errors related to `Cast(13)` and `tensor(int64)`. A typical error looks like this:

```text
Failed to find kernel for Cast(13) ... the node in the model has the following type (tensor(int64))
```

See the related ONNX Runtime issue:

- [microsoft/onnxruntime#25125](https://github.com/microsoft/onnxruntime/issues/25125)

The currently verified workaround is:

1. Download and use the [remove_back_to_back_cast.py](https://github.com/guschmue/ort-web-perf/blob/master/remove_back_to_back_cast.py) script 
2. Convert the original model with a command such as:

```powershell
python .\scripts\remove_back_to_back_cast.py .\imageseg-training\training_result\rtdetrv2_original_style_hugginface\model_fp16.onnx .\imageseg-training\training_result\rtdetrv2_original_style_hugginface\model_fp16_encoded.onnx
```

Then import the converted `model_fp16_encoded.onnx` into the Web App for inference.
